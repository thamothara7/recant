## 1. Eviction transport: poll the outbox, keep the bus interface

The W1 spike (docs/spike-changefeeds.md) already decided the Cloud path:
webhook changefeed into the fanout Lambda, with an outbox poller as fallback
behind one `EvictionBus` interface. W3 needs the local half of that decision.
Investigated 2026-07-10 against current docs; step 1 of the order of work
confirms empirically on the running cluster:

- Since the v24.3 licensing change there is one binary and no per-feature
  enterprise gating; enforcement is throttling. An unlicensed multi-node
  cluster gets a 7-day grace period, then is throttled to 5 concurrent open
  SQL transactions. Single-node clusters are exempt; our chaos cluster is
  3 nodes, so its license state is a real operational variable (section 8).
- Sinkless changefeeds (`CREATE CHANGEFEED FOR TABLE ...` with no `INTO`)
  stream rows over the SQL session; `kv.rangefeed.enabled` is already set by
  `ops/chaos/init.sh`.
- The webhook sink is HTTPS-only: a local webhook sink means a self-signed
  TLS endpoint plus `insecure_tls_skip_verify` on the sink URI, TLS cert
  bootstrap in the dev loop for no demo payoff.

**Decision: the W3 local transport is a plain polling consumer over the
`memory_events` outbox, not a local changefeed.** Three reasons: it is
unconditionally free and immune to the grace-period ambiguity on the 3-node
cluster; it is restart-safe with a durable per-consumer delivery row instead
of resolved-timestamp bookkeeping; and a local sinkless consumer would not
de-risk (b) anyway, because the Lambda parses the webhook envelope, a
different wire format from the sinkless SQL stream. Tradeoff recorded: we
give up push latency (worst case one poll interval, 250 ms default, still
sub-second for the proof-moment-4 rehearsal) and the local pipeline does not
exercise rangefeeds; the Cloud webhook changefeed remains the primary
transport per the spike and replaces only the poll loop when U1+U3 land.

**Decision: `memory_events` stays append-only; delivery state lives in a new
`fanout_deliveries` table, and the poll is an anti-join, not a timestamp
cursor.** The outbox is audit evidence (W4 forensics reads it), so the
consumer must not mutate it; and a timestamp cursor has the classic outbox
hazard where a transaction commits an event with `created_at` below an
already-advanced cursor and the event is skipped forever. The anti-join
cannot skip.

Migration 0006, part 1:

```sql
CREATE TABLE IF NOT EXISTS fanout_deliveries (
    event_id        UUID NOT NULL REFERENCES memory_events (event_id),
    consumer        STRING NOT NULL,
    delivered_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    evicted_rows    INT8 NOT NULL DEFAULT 0,
    aborted_actions INT8 NOT NULL DEFAULT 0,
    PRIMARY KEY (event_id, consumer)
);
```

Poll query shape (env `RECANT_FANOUT_POLL_MS`, default 250; consumer name is
a CLI flag so tests and a future second consumer never collide):

```sql
SELECT e.event_id, e.incident_id, e.payload
FROM memory_events e
LEFT JOIN fanout_deliveries d
       ON d.event_id = e.event_id AND d.consumer = %s
WHERE d.event_id IS NULL AND e.kind = 'recant'
ORDER BY e.created_at
LIMIT 50;
```

Cross-event ordering is best-effort (`created_at` can tie); harmless here
because evictions are idempotent deletes and commute.

## 2. Fanout: one handler module, two entrypoints

New top-level `fanout/` package (spec section 8; the Lambda + EventBridge IaC
lands here in (b)). The binding constraint: **the local worker and the Lambda
share one handler module.** Transport code stays thin enough that neither
entrypoint contains eviction logic.

- `fanout/handler.py`: transport-agnostic core, no AWS imports at module
  scope.
  - `parse_event(raw) -> RecantEvent`: validates the decision-12 contract
    (`payload.evictions = [{agent_id, belief_ids}]`), rejects malformed
    payloads loudly, ignores kinds other than `recant`.
  - `apply_evictions(conn, event) -> Receipt`: runs INSIDE the caller's
    transaction (same pattern as `compute_closure`), so apply and delivery
    commit atomically.
- `fanout/worker.py`: the local process. Poll loop; for each event, one
  serializable `run_txn`: `apply_evictions` + `INSERT INTO fanout_deliveries`.
  Exactly-once effect per consumer; a crash between apply and commit rolls
  back the whole thing and the next pass redelivers.
- `fanout/lambda_entry.py`: written NOW, unit-tested only (section 7 covers
  deploy). Webhook-sink envelope in, `parse_event` per row, EventBridge
  `PutEvents` batches out, boto3 client injected so tests need no AWS.

`apply_evictions`, per event, in order:

1. `DELETE FROM agent_memory WHERE belief_id = ANY(%s)` over the union of the
   event's eviction belief ids, `RETURNING agent_id, belief_id` for the
   receipt counts.
2. `UPDATE agent_actions SET status = 'aborted', status_reason = 'recant',
   incident_id = %s, resolved_at = now() WHERE status = 'pending' AND
   derived_from && %s RETURNING action_id, agent_id`. Array overlap catches
   pending actions no matter when they were enqueued relative to the recant.
3. Receipt event for the console ticker and W4 forensics:
   `INSERT INTO memory_events (kind = 'eviction', incident_id, payload)` with
   `{consumer, source_id, apply_ms, evictions: [{agent_id, belief_ids,
   evicted_rows}], aborted_actions: [{action_id, agent_id}]}`. The poller
   filters `kind = 'recant'`, so receipts cannot self-loop.
4. `INSERT INTO fanout_deliveries` with the counts.

**Decision: eviction deletes run as direct SQL, not through the vector
store's delete API.** The delete, the action abort, the receipt, and the
delivery row must commit in one serializable transaction, and the package's
session-managed API cannot join a caller-provided transaction. The direct-SQL
touchpoints on the package-owned table are pinned by the bootstrap shape test
(section 3) so a package upgrade fails loudly, not silently.

## 3. Agent working memory: langchain-cockroachdb

Package verification (2026-07-10): `langchain-cockroachdb` exists on PyPI,
published by Cockroach Labs (github.com/cockroachdb/langchain-cockroachdb).
It provides `CockroachDBEngine`, `CockroachDBVectorStore` (sync + async),
`CockroachDBChatMessageHistory`, a LangGraph checkpointer, C-SPANN vector
index helpers, metadata filtering, delete-by-ids, and built-in retry on
40001. Python 3.10+, so our 3.12 venv is fine.

**Decision: `langchain-cockroachdb`, not `langchain-postgres`.**
langchain-postgres assumes the pgvector extension (`CREATE EXTENSION vector`
and pgvector-specific DDL) which CockroachDB rejects; langchain-cockroachdb
is CRDB-native (VECTOR type, C-SPANN indexes) and ships its own
serialization-retry loop. It follows the langchain-postgres v2 engine API
shape, so the swap risk is contained if the package ever regresses.

**Decision: working memory is a materialized runtime copy keyed by
`belief_id`, not a read-time status filter on `beliefs`.** Eviction must be a
real operation (proof moment 4 evicts caches and aborts in-flight work, on
camera); filtering at read time would make the fanout a no-op and hide the
story. `beliefs` stays the custody record; `agent_memory` is the context
store; the `belief_id` key is the custody link between them.

Expected table shape. Creation is owned by
`CockroachDBEngine.init_vectorstore_table` in `fleet/bootstrap.py`; exact
column names are package-owned, so the shape below is pinned by a test, not
assumed:

```sql
-- expected shape of the package-created table (bootstrap shape test pins it)
agent_memory (
    langchain_id       STRING PRIMARY KEY,  -- set to str(belief_id): mirror is idempotent
    content            STRING NOT NULL,
    embedding          VECTOR(1024) NOT NULL,
    agent_id           UUID NOT NULL,       -- typed metadata column
    belief_id          UUID NOT NULL,       -- custody link into beliefs
    langchain_metadata JSONB
)
```

One table for all agents with an `agent_id` metadata column (single vector
index, one-statement eviction, per-agent retrieval via metadata filter)
rather than a table per agent. Bootstrap applies the package's vector index
helper with the COSINE strategy: the W2 opclass lesson (decision 8) says
never ship an L2 index under a cosine query, and the W6 seed-at-scale pass
will need the index; an EXPLAIN assertion mirrors the W2 test. The engine URL
is derived from `DATABASE_URL` by scheme swap only (`postgresql://` to the
package's `cockroachdb://`), one helper in bootstrap.

**Decision: the working-memory table is bootstrapped by the package, not
added to the numbered migration chain.** The migrations own the custody
substrate; `agent_memory` is a disposable runtime cache whose columns belong
to a third-party package, and freezing its DDL in our chain would break the
chain on any package upgrade. Bootstrap is idempotent and logged; the shape
test fails the suite if an upgrade moves a column the eviction SQL touches.
Fallback if the package fights local v26.2 at bootstrap (order-of-work step
3 is the go/no-go): own the table in migration 0007 with plain SQL retrieval
behind the same `FleetMemory` seam, log the cut in plan.md.

Chat history: `CockroachDBChatMessageHistory`, one session per agent; each
tick appends what the agent observed and decided. **Decision: eviction never
writes into an agent's transcript.** The transcript is the agent's own
record; forging entries from the fanout would corrupt the custody story. The
visible proof of eviction is the context-assembly diff (`fleet.show`, section
4) plus the receipt event.

Embeddings adapter: `fleet/embeddings.py` wraps the existing
`services.common.embedder.Embedder` protocol as a LangChain `Embeddings`
(`embed_documents` / `embed_query`). Embedder selection is env-driven
(`RECANT_EMBEDDER`, default `hash`); the adapter is embedder-agnostic, so
Titan lands in (b) with zero fleet changes.

## 4. Fleet: three scripted agents through the gateway

New top-level `fleet/` package (spec section 8). Deterministic and scripted,
no LLM calls in (a): spec section 10 demands a demo that needs no luck, and
Bedrock is behind U3 regardless. The tick loop IS the story generator.

- `fleet/gateway.py`: httpx client for the attest gateway
  (`RECANT_GATEWAY_URL`, default `http://localhost:8000`); tests use httpx's
  ASGITransport against the app in-process, same client code either way. The
  gateway remains the only write path into custody, even for the fleet.
- `fleet/agents.py`: researcher ingests source claims as beliefs; support
  retrieves similar context from its working memory and derives beliefs
  (explicit `parent_ids` where the script says so, the paraphrase with none:
  proof moment 2); ops retrieves policy beliefs and enqueues a pending
  `agent_actions` row `derived_from` those belief ids.
- `fleet/run.py`: `--ticks N` loop. Story constants (texts, controlled
  embeddings, edge script) are extracted from `ops/seed/seed.py` into one
  shared story module so seed and fleet cannot drift; `check_story()` still
  guards both directions before anything writes.
- `fleet/show.py`: prints an agent's assembled context (retrieved working
  memory plus recent chat turns) with belief ids and status. This is the
  before/after surface for demo-visible eviction.

Mirror rule: an agent mirrors a belief into working memory only after the
gateway returns 201, and only when the returned status is `active`, using
`str(belief_id)` as the vector-store document id so a retried mirror is an
upsert. **Decision: suspect-born beliefs never enter working memory.** The
gateway's birth-status (decision 11) closes the write side of post-recant
residue; the fleet honoring it closes the runtime side for writes that arrive
after the recant, while flip events (sections 1 and 2) evict what was
mirrored before it. **Also recorded, deliberately kept from W2: still no hard
gateway reject for writes citing recanted sources.** Suspect-plus-unmirrored
keeps the residue observable and honest; a hard 4xx would hide the very story
the demo tells. Revisit in W4 if the forensics API wants a policy switch.

Migration 0006, part 2:

```sql
CREATE TABLE IF NOT EXISTS agent_actions (
    action_id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    agent_id      UUID NOT NULL REFERENCES agents (agent_id),
    kind          STRING NOT NULL,              -- 'refund', 'publish', ...
    payload       JSONB NOT NULL,
    derived_from  UUID[] NOT NULL,              -- belief ids the action rests on
    status        STRING NOT NULL DEFAULT 'pending'
                  CHECK (status IN ('pending', 'executed', 'aborted')),
    status_reason STRING,
    incident_id   UUID REFERENCES incidents (incident_id),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    resolved_at   TIMESTAMPTZ,
    INDEX agent_actions_status_idx (status)
);
```

`derived_from` is `UUID[]` because CockroachDB has no FK-over-array; the
abort predicate is `derived_from && $evicted` scoped to `status = 'pending'`
(the status index keeps that scan small; an inverted index on `derived_from`
is a W6 scale item, not needed at demo cardinality).

The demo beat this section exists for: ops enqueues "auto-approve pending
365-day refund for customer #4471" derived from the support paraphrase;
`recant(forum_thread)` flips the paraphrase; the worker aborts the pending
action inside the same transaction that evicts the memory rows. That is proof
moment 4's local rehearsal, end to end, with no AWS.

## 8. Failure modes

- **Local license throttle.** Chaos-cluster volumes older than 7 days put the
  unlicensed 3-node cluster into throttling: 5 concurrent open SQL
  transactions, presenting as pool stalls rather than errors. Mitigation:
  scripted volume reset (`docker compose down -v`, init, migrate, seed,
  fleet run: minutes) or a free Enterprise Free license key; goes in the
  README failure-modes table. Single-node clusters are exempt but cannot run
  the node-kill beat.
- **Worker down.** Events accumulate in the append-only outbox and drain on
  restart via the anti-join: eviction delayed, never lost. Demo hygiene:
  start the worker before recording, check `fanout_deliveries` lag first.
- **Gateway 201 but mirror fails.** The belief is attested but absent from
  working memory: the safe direction (no unattested memory can ever exist).
  The next tick retries; the belief_id document id makes the retry an upsert.
- **Action executed before the recant.** An honest miss and the actual
  incident-response story: the receipt records only pending aborts; W4
  forensics shows what executed on a tainted basis and when.
- **Package upgrade moves `agent_memory` columns.** The bootstrap shape test
  fails the suite loudly; the eviction SQL and `fleet.show` are the only two
  direct-SQL touchpoints, both enumerated in that test.
- **Poll interval vs demo latency.** Worst-case visible eviction is one
  interval plus the apply transaction (receipt carries `apply_ms` for the
  judge overlay). If the recording needs tighter, lower the interval; do not
  switch transports for cosmetics.

## 10. Implementation findings (2026-07-10, on landing)

Recorded where reality diverged from the sketches above; decisions promoted to
plan.md log entries 18 through 21.

- **The license throttle fired for real.** Step 1's probe hit "No license
  installed. The maximum number of concurrently open transactions has been
  reached" on the 8-day-old volumes; `docker compose down -v` + init + migrate
  reset the grace period and 8/8 concurrent transactions passed. The sinkless
  changefeed opened and streamed silently until statement_timeout, as
  expected. Section 8's mitigation is now proven procedure.
- **langchain-cockroachdb 0.2.1 has a different table shape than sketched.**
  No typed metadata columns: the table is id (UUID PK), namespace column
  (TEXT), content, embedding, metadata JSONB, created_at. Mapping: id IS the
  belief id (upsert-by-id confirmed: INSERT ... ON CONFLICT DO UPDATE), the
  namespace column is agent_id. The shape test pins exactly these columns.
- **Two package workarounds, both load-bearing.** (1) The sync wrappers run
  each call under a fresh event loop; pooled asyncpg connections are bound to
  the loop that created them and die with "attached to a different loop" on
  the second call. Fix: NullPool via from_engine. (2) CSPANNIndex emits
  CREATE VECTOR INDEX with no opclass while COSINE search queries `<=>`: the
  package's own index cannot serve its own query on v26.2 (decision 8,
  again). Fix: bootstrap owns the DDL, a namespace-prefixed
  `(agent_id, embedding vector_cosine_ops)` index; EXPLAIN pins the
  `vector search` node against the package's exact query shape.
- **Engine URL**: scheme swap plus param strip:
  `postgresql://...?sslmode=disable` becomes `cockroachdb+asyncpg://...`
  (asyncpg rejects sslmode; TLS params return with U1). `uv add asyncpg` was
  required; the package does not depend on a driver.
- **Rehearsal timings** (for the judge overlay): recant SERIALIZABLE TXN 92ms
  with VECTOR kNN 16ms inside; eviction apply 31ms for 3 rows + 1 abort;
  worker cold start to backlog-drained just over 1s including interpreter
  startup.
